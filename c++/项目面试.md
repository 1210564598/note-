# 屏幕



##### 视觉检测基本流程

> ### **图像预处理**
>
> - **目的**：增强目标特征，降低噪声，提高检测稳定性
> - **常用方法**：
>   - 灰度化：将彩色图像转换为灰度
>   - 去噪：高斯滤波、均值滤波
>   - 对比度增强：直方图均衡化
>   - 二值化：阈值分割（THRESH_BINARY / THRESH_BINARY_INV）
>   - 边缘增强：Sobel、Laplacian







> ## **1. 滤波去噪**
>
> 在图像采集时，会受到传感器、环境光、机械抖动等因素影响，所以第一步常常要“去噪”。
>
> - **高斯滤波**
>   - 作用：用一个高斯核对邻域像素加权平均，抑制高斯噪声。
>   - 特点：平滑图像，保留整体轮廓。
>   - 应用：工业相机拍摄金属表面时，传感器会带来高斯噪声，可以用高斯滤波清理。
> - **中值滤波**
>   - 作用：取邻域像素的中值代替中心点。
>   - 特点：对“椒盐噪声”（黑白点）非常有效。
>   - 应用：震动或灰尘导致的随机点状噪声。
> - **形态学滤波**
>   - 操作：腐蚀（去掉小亮点）、膨胀（填补小暗点）、开运算(腐蚀-膨胀)、闭运算（线膨胀后腐蚀）。
>   - 开运算（去除小的噪声点，而不影响原来的图像）
>   - 闭运算（作用是消除 “闭合” 物体里面的孔洞）
>   - 应用：去除表面孤立小噪声，或者填补裂纹内部的小空洞。
>
> ------
>
> ## **2. 图像增强**
>
> 在缺陷检测中，缺陷往往很微弱，所以要增强对比度或锐化细节。
>
>  **直方图均衡化（Histogram Equalization）** 
>
>  普通均衡化 
>
> 优点：让灰度分布更均匀，对比度增强
>
> 缺点：在光照不均的图像上效果不好 
>
> 
>
> - **对比度增强**
>   - 方法：直方图均衡化。
>   - 全局直方图均衡化：整体提升亮暗对比度。
>   - 自适应直方图均衡化（CLAHE）：在局部区域调整对比度，更适合光照不均。
> - **锐化处理**
>   - 方法：利用拉普拉斯算子、高通滤波，增强边缘。
>   - 应用：让划痕、裂缝的边缘更加清晰。
>
> ------
>
> # 🔹 二、缺陷分割算法
>
> 分割就是把“缺陷区域”和“背景”区分开。
>
> ## **1. 阈值分割**
>
> - **全局阈值分割**
>   - 方法：Otsu 大津法，自动找到最佳阈值，把前景和背景分开。
>   - 应用：钢板表面划痕检测（划痕和背景灰度差大）。
> - **局部阈值分割**
>   - 方法：对每个小块区域单独设定阈值。
>   - 应用：光照不均匀的表面，比如弯曲金属。
>
> ------
>
> ## **2. 边缘检测**
>
> - **Canny 算法**
>   - 流程：高斯滤波 → 计算梯度 → 非极大值抑制 → 双阈值检测。
>   - 特点：边缘检测精度高，能提取裂纹边缘。
> - **Sobel/Prewitt 算子**
>   - 原理：计算水平方向、垂直方向的梯度。
>   - 应用：低对比度情况下的划痕检测。
>
> ------
>
> ## **3. 区域生长**
>
> - **原理**：从一个“种子点”开始，把与它灰度/颜色相近的像素合并成区域。
> - **应用**：适合复杂、不规则的缺陷（比如非均匀的划痕）。
>
> ------
>
> # 🔹 三、特征提取与分类算法
>
> 分割出缺陷后，还要判断“缺陷的类型、性质、严重程度”。
>
> ## **1. 几何特征**
>
> - 特征：面积、周长、圆度、长宽比。
> - 应用：
>   - 圆形缺陷 → 气泡、气孔
>   - 长条形缺陷 → 裂纹、划痕
>
> ------
>
> ## **2. 纹理特征**
>
> - **GLCM（灰度共生矩阵）**
>   - 能提取对比度、能量、熵、相关性等特征。
>   - 应用：布料表面破损、纸张纹理异常。
>
> ------
>
> ## **3. 深度学习方法**
>
> - **CNN（卷积神经网络）**
>   - 输入图像，直接输出缺陷类别（分类）或位置（检测）。
>   - 应用：用 YOLO/SSD 可以实时检测多种缺陷。
> - **U-Net**
>   - 网络结构：U 型，编码-解码，支持像素级分割。
>   - 应用：医学病灶分割（肺结节、肿瘤）、晶圆微小缺陷检测。
>
> ------
>
> # 🔹 总结流程
>
> 一个完整的缺陷检测通常是：
>
> 👉 **图像采集 → 预处理（去噪/增强） → 缺陷分割（阈值/边缘/区域生长） → 特征提取（几何/纹理） → 缺陷分类（机器学习/深度学习）**
>

>
> 在该视觉检测系统中，我负责整个**图像处理与缺陷识别核心环节**，具体职责如下：
>
> 1. **图像预处理与增强**
>    - 接收相机控制模块传来的原始图像数据
>    - 对图像进行**高斯滤波**去噪，同时保留边缘信息
>    - 使用**直方图均衡化**提升图像对比度，使亮部和暗部细节更加清晰
>    - 输出一副噪声更少、亮度分布均匀、对比度增强的“干净”图像，为后续缺陷检测提供可靠基础
> 2. **特征提取与结构化信息生成**
>    - 基于增强后的图像进行**关键点检测**，提取图像特征点并计算其描述符
>    - 使用**形态学开运算**优化候选区域，提取**结构化坐标信息**（ROI）、**Homography矩阵**以及**轮廓信息（Contours）**
>    - 结合原始图像与上述位置信息，精准锁定待检测的Die图像区域，为差分分析和缺陷定位提供支持
> 3. **差分分析与初步缺陷检测**
>    - 对目标Die区域进行差分法处理，生成**差异图**
>    - 对二值化后的差异图执行**Blob分析**，计算每个连通域的几何特征（如面积、周长等）
>    - 使用**Canny边缘检测**突出显示图像中的边缘结构，用于识别划痕、崩边等线性缺陷
>    - 将分析结果组织成**结构化缺陷候选列表**
> 4. **特征量化与缺陷分类**
>    - 对每个缺陷候选区域提取高级特征，包括几何特征、纹理特征、边缘特征等
>    - 使用基于OpenCV的**SVM（cv::ml::SVM）分类器**进行自动分类
>    - 输出每个Die的缺陷类型、位置、大小等信息，形成完整的缺陷报告
> 5. **系统集成与数据交互**
>    - 将处理结果和缺陷报告传递给**UI模块**显示
>    - 将数据写入**数据管理模块**进行存储和记录
>    - 向**打点器控制模块**下发系统指令，实现产线缺陷标记和反馈
>
> **总结**：我负责将原始图像数据转化为结构化、可分析的信息，并通过差分分析、边缘检测、特征量化和机器学习分类，完成从图像预处理到缺陷定位和分类的全流程，为系统生成可靠的缺陷报告并驱动后续产线操作提供支持。

##### 多光源检测流程

> ------
>
> ## **1. 暗场检测 detectDarkField**
>
> ```cpp
> cv::cvtColor(darkImg, gray, cv::COLOR_BGR2GRAY);
> cv::morphologyEx(gray, gray, cv::MORPH_TOPHAT, cv::Mat::ones(3,3,CV_8U));
> cv::Canny(gray, edges, 50, 150);
> ```
>
> - **灰度化**：减少数据量，只保留亮度信息，方便后续运算。
> - **顶帽变换 (Top-Hat)**：可以突出图像中**比周围更亮的细小结构**，这里用来增强 **划痕/压痕线条**。
> - **Canny 边缘检测**：把这些细小划痕提取出来，得到二值边缘图。
>
> ```cpp
> cv::absdiff(gray, grayBright, diff);
> cv::threshold(diff, diff, 20, 255, cv::THRESH_BINARY);
> cv::medianBlur(diff, diff, 3);
> edges |= diff;
> ```
>
> - **暗场 - 明场差分**：利用光源差异，突出压痕（凹陷在暗场更亮，在明场更暗）。
> - **阈值化 + 中值滤波**：去掉噪声，保留真正的压痕区域。
> - **按位或（|=）**：把划痕和压痕的检测结果合并。
>
> ✅ **解决的问题**：小划痕/压痕往往不明显，单用一个算子容易漏检，通过 **形态学增强 + 差分**，提高检测的灵敏度。
>
> ------
>
> ## **2. 明场检测 detectBrightField**
>
> ```cpp
> cv::blur(gray, mean, cv::Size(7,7));
> cv::blur(gray.mul(gray), sqmean, cv::Size(7,7));
> cv::sqrt(sqmean - mean.mul(mean), varImg);
> cv::threshold(varImg, mask, 15, 255, cv::THRESH_BINARY);
> ```
>
> - **局部方差计算**：计算图像每个小区域的亮度变化程度。
>   - 如果区域比较均匀 → 方差小
>   - 如果有脏污/指纹 → 亮度变化剧烈 → 方差大
> - **阈值分割**：把高方差区域标记为缺陷。
>
> ✅ **解决的问题**：污点/指纹/水渍不像划痕那样有明显边缘，而是局部亮度分布异常，用 **方差检测**能把这些不规则区域找出来。
>
> ------
>
> ## **3. 背光检测 detectBackLight**
>
> ```cpp
> cv::threshold(gray, thr, 100, 255, cv::THRESH_BINARY_INV);
> mask = thr;
> cv::Canny(gray, edges, 50, 150);
> mask |= edges;
> ```
>
> - **阈值反选 (THRESH_BINARY_INV)**：在背光下，气泡/残胶会呈现黑斑，反选后这些黑斑就被提取出来。
> - **Canny 边缘检测**：再额外检测边缘轮廓，发现断边、崩边。
> - **合并结果 (|=)**：得到气泡/残胶 + 边缘缺陷的综合掩码。
>
> ✅ **解决的问题**：背光下能很；好地凸显透光异常（气泡、残胶），但边缘崩缺需要额外检测 → **阈值 + 边缘检测结合**才完整。
>
> ------
>
> ## **4. 合并缺陷掩码**
>
> ```cpp
> cv::Mat mask_final = mask_dark | mask_bright | mask_back;
> ```
>
> - **按位或 (bitwise OR)**：把三种光源检测到的缺陷整合在一起。
> - 优点：
>   - **全面性**：保证任何光源下能看到的缺陷都会被计入。
>   - **避免漏检**：比如有的缺陷在暗场看不清，但在明场或背光下明显。
>
> ✅ **解决的问题**：单一光源检测不够可靠，多光源检测结果融合提高了**检测完整性**。
>
> ------
>
> ## **5. 可视化 displayResult**

##### 图片预处理

> **高斯滤波（Gaussian Blur）**
>
> - 作用：去除图像中的高频噪声，同时**尽量保留边缘**。
> - 为什么重要：实际产线拍摄图像会有传感器噪声或微小灰尘，直接做边缘检测或方差分析可能会误报。高斯滤波可以平滑噪声，让后续算法更稳定。
>
> **直方图均衡化（Histogram Equalization）**
>
> - 作用：调整图像亮度分布，让暗部和亮部细节都更清楚，整体对比度增强。
> - 为什么重要：暗场、明场和背光图像的亮度可能不均匀，如果不做均衡化，划痕、污点或气泡的对比度不够，容易被漏检。
>
> **输出“干净”图像**
>
> - 经过高斯滤波和直方图均衡化后的图像，噪声更少、亮度分布均匀，**亮部和暗部细节更清晰**。
> - 后续检测步骤（Canny 边缘、局部方差、阈值分割、差分分析）都基于这个增强后的图像，能显著提高缺陷检测的准确率和稳定性。
> - 
>
> 这一步是视觉检测流程的**必备预处理**，目的是让后续的算法更可靠：
>
> - 划痕/压痕：Canny 边缘检测对噪声敏感，必须用高斯滤波降噪
> - 污点/指纹：局部亮度方差依赖对比度均衡
> - 气泡/残胶和边缘缺陷：阈值分割更稳健
>
> 所以，这些操作不仅“需要”，而且是整个检测流程的**基础保障**。
>
> ```c++
> cv::Mat gray, blurred, equalized;
> 
> // 转灰度
> cv::cvtColor(src, gray, cv::COLOR_BGR2GRAY);
> 
> // 高斯滤波（核大小 3~5，根据分辨率调整）
> cv::GaussianBlur(gray, blurred, cv::Size(3,3), 0);
> 
> // 直方图均衡化
> cv::equalizeHist(blurred, equalized);
> 
> // equalized 就是“干净图像”，后续缺陷检测使用它
> 
> ```

# 注塑机







##### 运行过程



> 注塑机的工作过程其实就是把塑料颗粒加热融化，然后高压注入模具里，冷却成型，最后把制品顶出来。
>
> 具体来说，它大概分几个主要步骤：
>
> > 1. **加料**：先把塑料颗粒从料斗送到加热的料筒里。
> > 2. **塑化**：料筒外面有加热圈，同时里面有螺杆在旋转，把塑料加热到熔融状态，并且搅拌均匀。
> > 3. **注射**：螺杆向前推进，把融化的塑料高压打进模具型腔里。
> > 4. **保压**：模具里充满后，继续保持一段压力，避免制品缩水或有气孔。
> > 5. **冷却**：保持模具闭合，等制品在模具里冷却、硬化。
> > 6. **开模/脱模**：模具打开，顶针把制品顶出来。
> >
> > 这个过程是循环往复的：合模 → 注射 → 保压 → 冷却 → 开模 → 顶出 → 再合模。





### 

> ### **1️⃣ 合模（Closing）**
>
> **作用**：将模具完全闭合并锁紧，保证注射过程中模具不会被熔融塑料的压力顶开。
>
> **控制重点**：
>
> - 锁模力（Locking Force）足够大，避免安全事故。
> - 模具位置精确，保证模具闭合面平整。
>
> **上位机实现**：
>
> - 上位机通过 EtherCAT 或 PLC 控制伺服电机或液压系统，精确控制滑块移动。
> - 实时采集锁模力传感器和位置传感器数据，绘制曲线供操作员参考。
> - 如果锁模力或位置异常，上位机会触发报警，防止下一步注射。
>
> ------
>
> ### **2️⃣ 注射（Injection）**
>
> **作用**：将熔融塑料注入模具腔体，形成产品的形状。
>
> **控制重点**：
>
> - 注射速度（Injection Speed）：太快可能产生飞边，太慢会填充不满。
> - 注射压力（Injection Pressure）：保证材料充分填充模具，避免气泡或空洞。
> - 螺杆位置（Screw Position）：保证注射量准确。
>
> **上位机实现**：
>
> - 上位机通过运动控制模块向伺服电机发送速度和位置指令。
> - 实时采集压力传感器和螺杆位置数据，通过 PID 或自适应控制算法进行闭环控制。
> - 数据显示在界面上，压力和位置曲线实时更新，方便操作员监控。
>
> ------
>
> ### **3️⃣ 保压（Packing）**
>
> **作用**：在注射完成后继续施加压力，补偿塑料冷却收缩，保证制品密度和外观。
>
> **控制重点**：
>
> - 保压压力和时间：保证制品厚度和尺寸精度。
> - 避免过高压力导致闪边或模具损坏。
>
> **上位机实现**：
>
> - 上位机将保压压力指令发送到控制器，驱动伺服或液压电机保持压力。
> - 实时采集压力数据，通过 PID 控制精确维持压力。
> - 如果压力超出设定范围，系统自动报警或调整。
>
> ------
>
> ### **4️⃣ 冷却（Cooling）**
>
> **作用**：让塑料在模具中冷却、固化，保证尺寸稳定性和机械性能。
>
> **控制重点**：
>
> - 温度均匀：不同部位冷却不均会导致翘曲或缩痕。
> - 冷却时间：太短制品不成型，太长影响生产效率。
>
> **上位机实现**：
>
> - 上位机通过多通道 PID 控制模具加热/冷却系统（温控器、冷却水路等）。
> - 实时采集各通道温度传感器数据，绘制温度曲线。
> - 通过历史数据和报警系统，确保温度在目标范围内。
>
> ------
>
> ### **5️⃣ 开模和顶出（Ejection）**
>
> **作用**：打开模具，使用顶针将制品推出模具，完成一个生产循环。
>
> **控制重点**：
>
> - 开模位置和速度：保证制品完整，不碰撞模具。
> - 顶出动作力度：避免制品损坏。
>
> **上位机实现**：
>
> - 上位机通过运动控制模块发送开模和顶出指令。
> - 实时监控滑块位置、顶针位移、传感器信号，确保动作完成。
> - 操作界面显示本次循环状态，并记录异常数据。
>
> ------
>
> ### **整体控制特点**
>
> 1. **闭环控制**：温度、压力、位置都是实时采集反馈，保证精度。
> 2. **报警机制**：任何异常（超压、超温、位置异常）都会触发报警并停止动作。
> 3. **数据可视化与记录**：上位机绘制实时曲线、存储历史数据，便于分析和追溯。

### 温度控制

>  这个温度控制模块，主要是用来精确控制注塑机各个加热区的温度,

> 1. **初始化阶段**：
>    - 首先，我们创建 `DataManager` 对象，传入串口设备路径和每个传感器的温度、压力上下限。
>    - 构造函数会建立 Modbus RTU 通信，设置好波特率、校验、超时时间，保证能跟传感器正常通信。
> 2. **采集数据**：
>    - 启动 `startAcquisition()` 后，它会进入一个循环，每 50ms 采集一次数据。
>    - 每轮循环会生成一个 `SensorData` 结构体，把两个温度传感器和一个压力传感器的读数都存进去，并记录当前时间。
> 3. **历史缓存**：
>    - 采集完成后，会把 `SensorData`，将数据存入vector展示到ui，再将数据存入队列中，用于日志编写。
> 4. **实时异常判断**：
>    - 每次采集完数据，就会马上调用 `checkThresholds()` 检查温度或压力是否超过上下限。
>    - 如果超出，就触发回调函数，把这条数据和报警类型一起发送出去。
>    - 这样报警模块、UI、日志模块都能第一时间得到具体的异常数据，包括数值和时间。
> 5. **循环执行**：
>    -  “我在一个循环里，每轮循环先向所有传感器寄存器发送 Modbus 读取请求，获取最新温度和压力。读取完成后，把数据存入队列或 map，同时做阈值判断。然后休眠 50 毫秒，再进入下一轮循环，保证每 50ms 就能获取一次数据。”

##### can总线

> 我来帮你详细梳理一下 **CAN 总线（Controller Area Network）**，这在工业控制、汽车电子里用得非常多，面试问到也很常见。
>
> ------
>
> ## 🔹 什么是 CAN 总线
>
> - **定义**：CAN（Controller Area Network）是一种 **多主机（multi-master）、消息广播型、基于优先级仲裁** 的串行通信总线。
> - **起源**：最早由 Bosch 公司为汽车电子系统设计，用来在发动机、变速箱、ABS 等 ECU 之间通信。
> - **应用领域**：汽车、工业自动化、半导体设备、医疗设备等。
>
> ------
>
> ## 🔹 CAN 总线的特点
>
> 1. **多主机通信**
>    - 总线上所有节点地位平等，都可以主动发送消息。
>    - 不像 Modbus 那样是主从结构。
> 2. **基于 ID 的仲裁机制**
>    - 每条消息都有一个 **ID（标识符）**，ID 越小，优先级越高。
>    - 当多个节点同时发送时，会自动仲裁，保证高优先级消息不会丢。
> 3. **高可靠性**
>    - 带有 **CRC 校验**、**错误检测**、**自动重发**机制，适合噪声环境。
>    - 常用于汽车、工厂等强干扰场合。
> 4. **速率灵活**
>    - 典型速率：10 kbps ~ 1 Mbps（经典 CAN），CAN FD 可到 5~8 Mbps。
>    - 通信距离和速率成反比（速率越高，距离越短）。
>
> ------
>
> ## 🔹 CAN 报文结构
>
> 1. **帧类型**
>    - **数据帧**：传输实际数据（0~8 字节，CAN FD 可到 64 字节）
>    - **远程帧**：请求数据
>    - **错误帧**：检测到错误时发送
>    - **过载帧**：延迟一段时间再发送
> 2. **典型报文格式**（标准帧 11 位 ID）：
>    - **ID（标识符）**：决定优先级
>    - **RTR**：远程帧标志
>    - **DLC（Data Length Code）**：数据长度（0~8）
>    - **Data**：数据区
>    - **CRC**：校验
>
> ------
>
> ## 🔹 CAN 总线的应用场景
>
> 1. **汽车电子**
>    - 发动机控制单元、车身控制系统（车窗、灯光）、ABS、仪表盘等。
>    - 节点多、实时性要求高、抗干扰强。
> 2. **工业设备**
>    - 半导体设备、机械手臂、变频器通信。
>    - 常见协议：CANopen、DeviceNet。
> 3. **电梯、医疗设备**
>    - 多控制单元之间的安全可靠通信。
>
> ------
>
> ## 🔹 项目中的口语化描述（面试用）
>
> > “在项目中，我们用 CAN 总线来做多模块通信，比如上位机和运动控制器之间的参数传输。
> >  我主要通过驱动库提供的 API，比如 `canInitialize(channel, baudRate)` 初始化通道，
> >  用 `canWrite(channel, id, data, dlc)` 发送数据帧，
> >  再用 `canRead(channel, &id, data, &dlc)` 轮询或回调方式接收数据。
> >
> > 因为 CAN 总线是基于 ID 的仲裁，所以我们在设计通信协议时，会分配不同 ID 给不同功能模块，比如 0x100 是温度传感器，0x200 是电机控制器。这样保证了实时性和可靠性。”
>
> ------
>

##### 日志轮转

> 在日志模块的实现里，除了基本的日志写入，还要考虑 **日志文件过大** 的问题，所以需要加上日志轮转。整体过程是这样的：
>
> 1. **初始化时设置轮转规则**：系统启动时，在 `Logger` 类里会设定一个最大日志文件大小，比如 10MB，超过这个大小就触发日志轮转。
> 2. **每次写日志时检查文件大小**：在 `logInfo()`、`logWarning()`、`logError()` 等方法写入文件之前，日志模块会检查当前日志文件的大小。
> 3. **触发轮转**：如果日志文件超过设定的最大值，就会执行轮转逻辑：
>    - 先关闭当前日志文件
>    - 把它重命名为带时间戳或序号的历史文件，比如 `system_20250902.log`
>    - 再重新生成一个新的 `system.log` 文件，用来继续记录新的日志
> 4. **保证历史日志可追溯**：通过日志轮转，我们既能保证当前日志文件不会无限增大影响性能，也能保留历史日志文件用于长期故障排查。
> 5. **结合报警日志**：报警日志依然会同步写入数据库，这部分不受轮转影响，因为数据库存储和文件日志是两条并行的链路。



##### 日志线程安全

> 其实日志线程安全最核心的点，就在于多个线程可能同时写日志，如果大家一起往同一个文件里写，就会出现日志内容交叉、乱序，甚至文件损坏。
>
> 我这边用的思路是 生产者-消费者模型。业务线程只管“生产日志”，就是把一条条日志格式化好，放到一个安全的队列里，然后立刻返回，不直接写文件。这样业务逻辑就不会因为 I/O 阻塞而卡住。
>
> 在后台，我们开一个专门的日志线程，它就是“消费者”。这个线程会不断从队列里拿出日志，然后统一写到文件里，还负责检查日志文件大小，如果超过上限就做一次日志轮转，把旧文件归档起来，开一个新的文件继续写。
>
> 这里的关键就是线程之间怎么安全地访问队列。我用了互斥锁 `mutex` 来保护队列操作，每次入队或者出队都先加锁，保证不会两个线程同时改队列。同时配合条件变量 `condition_variable`，避免后台线程空转，它会在没有日志的时候阻塞，直到有新日志进来才被唤醒。
>
> 文件写这块我也做了限制：只有后台线程能写文件，其它线程根本不碰文件，所以就没有并发写的问题了。



#####  **避免 CPU 空转** 

> cv.notify_one();  // 通知消费者
>
> cv.wait(lock, [] { return !q.empty(); }); // 等待，直到队列非空
>
>  日志模块里，`condition_variable` 主要用来实现 **生产者-消费者模型**，避免日志线程空转。 
>
> - 业务线程是“生产者”，它在调用 `logInfo()`、`logError()` 等方法时，把日志消息放进一个队列里。放完以后，会调用 `cv.notify_one()`，通知后台线程有新日志了。
> - 后台日志线程是“消费者”，它一直在cv.wait()阻塞等待。这个 `wait` 会自动释放锁，并把线程挂起。
> - 当有新日志进队列时，后台线程被唤醒，重新拿到锁，从队列里取出日志，写到文件里。
>
> 这样就避免了后台线程用 `while(true)` 死循环疯狂检查队列是否为空的问题，节省了 CPU 资源。
>
> 
>
>  一个线程在“生产东西”（比如写日志时把一条消息放进队列）。
>  另一个线程在“消费东西”（比如日志线程要从队列里取出消息写到文件）。
>
> 如果消费者线程不停地 while 循环检查队列是不是空，就会白白浪费 CPU 
>
> 这时候 **`condition_variable`（条件变量）** 就派上用场了。
>
> 它的作用很像一个“门铃”或者“信号器”：
>
> - 消费者线程说：“我现在没活干，就在门口等着（wait），直到有人按门铃叫我。”
> - 生产者线程在放入新日志后，会按一下门铃（`notify_one()` 或 `notify_all()`），告诉消费者：“有新活了，快起来干活！”
> - 消费者线程被唤醒，再去拿队列里的数据，继续写文件。
>
> 所以简单说：
>
> - `condition_variable` 用来 **让一个线程睡眠，直到另一个线程通知它**，避免 CPU 空转。
> - 它通常和 `std::mutex`、`std::unique_lock` 搭配使用，保证等待和唤醒之间的操作安全。





### 数据库

> “数据库管理模块的主要作用是把采集到的数据、报警信息和日志信息安全、可靠地存储到数据库中，方便后续查询、分析和报表生成。它需要和数据采集模块、日志模块以及报警模块对接，实现统一存储和管理。”
>
> 具体包括：
>
> 1. **存储采集数据**
>    - 每条采集的数据（温度、压力、时间戳等）写入数据库表，例如 `sensor_data`。
>    - 数据量大，可以采用批量提交（transaction）提高写入效率。
> 2. **存储报警信息**
>    - 当数据异常时，将报警信息（类型、数值、时间）写入 `alarm_log` 表。
>    - 提供查询接口，用于故障排查。
> 3. **存储日志信息**
>    - 系统运行日志（启动、停止、操作事件）写入 `system_log` 表。
>    - 支持按时间、级别查询。
> 4. **数据库访问封装**
>    - 提供统一的接口类，例如 `DatabaseManager`。
>    - 提供方法：`insertSensorData()`, `insertAlarm()`, `insertLog()`，内部使用事务管理，提高可靠性。
> 5. **线程安全**
>    - 数据库操作可能来自不同线程（采集线程、日志线程），需要加锁或使用线程安全的队列+单独数据库写入线程。

##### 数据库表

> 1. 工艺参数表 (主键，批次号，目标温度，目标压力，冷却时间，创建时间)
> 2. 运行数据表(主键，批次号，通道编号，实际温度，实际压力，记录时间)
> 3. 报警日志(主键，报警代码，报警信息描述，报警级别，报警发生时间)
> 4. 用户表（主键，用户名，密码，角色，角色创建时间）
> 5. 操作日志表（主键，操作用户，执行动作，动作详情，操作时间，外键关联用户表）



#####  为什么要在数据库里存储工艺参数和运行数据？ 

> 工艺参数是生产过程的基准，需要保存以便下次复现或调整；
>
> 运行数据（温度、压力曲线）可以用于质量追溯、工艺优化和故障分析；
>
> 报警数据有助于维护和安全管理。
>  👉 强调“数据驱动生产”和“质量追溯”的价值。



#####  在高并发情况下，如何保证数据写入的性能和一致性？ 

> 采用 **批量插入**，减少频繁的 I/O；
>
> 使用 **事务** 确保写入的一致性；
>
> 加上 **互斥锁或线程安全队列** 保证多线程并发安全；
>
> 可以根据实际需求，部分历史数据写入 **缓存/队列** 再异步落盘。



#####  如果系统突然断电，如何保证数据不丢失？ 

> 使用 **事务机制**，确保数据要么写入成功，要么回滚；
>
> 关键数据采用 **周期性落盘 + 缓存刷写策略**；
>
> 



#####  数据库查询效率怎么优化？ 

> 建立合理的 **索引**（如时间戳、工艺编号）；
>
> 使用 **分页查询** 避免一次性加载大数据量；
>
> 对历史数据进行 **归档**，减少主库压力；
>
> 可以用 **缓存** 提升热点数据访问速度。



#####  为什么要设计独立的接口方法 

> 保证 **接口清晰、职责单一**，便于维护和扩展；
>
> 不同业务逻辑分开处理，提高代码可读性；
>
> 方便未来在不改上层逻辑的情况下切换数据库或优化存储实现。

>   2. **串口**（RS232/RS485，Modbus RTU）

##### 单双轴的实现区别

> 很好的问题！我们可以用**直观的方式**来看“双轴运动”和“单轴运动”的区别：
>
> ------
>
> ## 1️⃣ 轴的数量
>
> - **单轴**：只控制一个电机（一个坐标方向），比如 X 轴的直线移动。
> - **双轴**：同时控制两个电机（两个坐标方向，例如 X + Y），实现平面运动。
>
> 口语理解：
>
> > 单轴就像一条直线跑道，你只能前后跑；
> >  双轴就像在平面跑步，你可以同时前后和左右跑，跑出直线、曲线甚至图形。
>
> ------
>
> ## 2️⃣ 控制逻辑
>
> - **单轴**：只需要告诉电机目标位置、速度、加速度就行。
> - **双轴**：
>   - 需要**轴组 (AxesGroup)** 把两个轴“捆绑”在一起。
>   - 运动时要**同步控制**，保证 X 和 Y 同步到达目标位置。
>   - 常用功能块是 `MC_MoveLinearAbsolutePreparation` + `MC_MovePath`。
>
> 口语理解：
>
> > 单轴就像一个人自己跑；
> >  双轴就像两个人拉着绳子一起跑，你必须保证他们步伐一致，否则会跑歪。
>
> ------
>
> ## 3️⃣ 路径规划
>
> - **单轴**：路径就是一个数值 → 目标位置
> - **双轴**：路径是一个向量 `[X, Y]`，可能还有多段直线、曲线
> - 双轴运动常用插补（线性插补、圆弧插补），保证移动轨迹平滑
>
> 口语理解：
>
> > 单轴只看一个数，去哪里就去哪里；
> >  双轴要看两个数，像画图一样，要沿直线或曲线走过去。
>
> ------
>
> ## 4️⃣ 调试与监控
>
> - **单轴**：监控一个轴的速度、位置即可
> - **双轴**：要监控两个轴，同时看它们的坐标是否同步
> - 错误处理也更复杂，如果一个轴出错，另一轴可能需要停止
>
> 口语理解：
>
> > 单轴就像你开一辆单车；
> >  双轴就像开一辆双轮车，两个轮子要同步，才能直行。
>

>
> - TwinCAT 提供 **PLCopen Motion Control 库**，功能块包括：

>

> ```c++
> (*
>     双轴插补运动控制示例
>     目标：在 XY 平面上移动到 (X=200mm, Y=100mm)，速度 100mm/s
> *)
> 
> PROGRAM MAIN
> VAR
>     // === 轴与轴组引用 ===
>     Axis1 : AXIS_REF;           // 轴1 (X轴)，需要在 I/O 映射里绑定到具体驱动器

>     AxesGroup : AXES_GROUP_REF; // 轴组引用，用于插补控制，绑定到 Motion 中的 Group
> 
>     // === 功能块实例 ===
>     fbPower1 : MC_Power;  // 功能块：控制轴1上电/使能
>     fbPower2 : MC_Power;  // 功能块：控制轴2上电/使能
>     fbHome1  : MC_Home;   // 功能块：轴1回零
>     fbHome2  : MC_Home;   // 功能块：轴2回零
> 
>     fbAddAxis1 : MC_AddAxisToGroup;   // 功能块：将轴1加入到轴组
>     fbAddAxis2 : MC_AddAxisToGroup;   // 功能块：将轴2加入到轴组
>     fbEnableGroup : MC_GroupEnable;   // 功能块：使能轴组
> 
>     fbPrepLin : MC_MoveLinearAbsolutePreparation; // 功能块：准备一个直线段
>     fbMovePath : MC_MovePath;                    // 功能块：执行路径
> 
>     // === 控制变量（可通过在线调试或HMI操作） ===
>     bEnableAxis : BOOL := FALSE;   // 置 TRUE 时，上电所有轴
>     bHome : BOOL := FALSE;         // 置 TRUE 时，触发回零
>     bAddAxes : BOOL := FALSE;      // 置 TRUE 时，将轴加入组
>     bEnableGroup : BOOL := FALSE;  // 置 TRUE 时，使能组
>     bPrepare : BOOL := FALSE;      // 置 TRUE 时，准备路径
>     bExecute : BOOL := FALSE;      // 置 TRUE 时，执行路径
> 
>     // === 路径数据对象 ===
>     Buffer : ARRAY[1..2048] OF BYTE;   // 存放路径数据的字节缓存
>     Path   : MC_PATH_DATA_REF(ADR(Buffer), SIZEOF(Buffer)); // 创建 Path 对象并绑定到缓存
> 
>     // === 目标参数 ===
>     aTargetPos : ARRAY[1..2] OF LREAL; // 目标位置数组，按顺序存放 [X, Y]
>     fVelocity  : LREAL := 100.0;       // 路径速度，单位：mm/s
>     fAccel     : LREAL := 200.0;       // 加速度
>     fDecel     : LREAL := 200.0;       // 减速度
> END_VAR
> 
> // ========== 程序循环 ==========
> 
> // 1) 上电两个轴
> fbPower1(
>     Axis := Axis1,    // 轴引用
>     Enable := bEnableAxis   // TRUE = 上电，FALSE = 断电
> );
> 
> fbPower2(
>     Axis := Axis2,    // 轴引用
>     Enable := bEnableAxis
> );
> 
> // 2) 回零
> fbHome1(
>     Axis := Axis1,    // 回零的轴
>     Execute := bHome  // TRUE = 开始回零，完成后 Done = TRUE
> );
> 
> fbHome2(
>     Axis := Axis2,
>     Execute := bHome
> );
> 
> // 3) 将两个轴加入到轴组
> fbAddAxis1(
>     GroupIn := AxesGroup,  // 目标轴组
>     AxisIn := Axis1,       // 要加入的轴
>     Execute := bAddAxes    // TRUE 时执行一次，Done = TRUE 表示完成
> );
> 
> fbAddAxis2(
>     GroupIn := AxesGroup,
>     AxisIn := Axis2,
>     Execute := bAddAxes
> );
> 
> // 4) 使能轴组
> fbEnableGroup(
>     AxesGroup := AxesGroup,   // 需要使能的轴组
>     Execute := bEnableGroup   // TRUE 时执行一次
> );
> 
> // 5) 准备直线段路径
> IF bPrepare THEN
>     Path.ClearPath(); // 清空旧路径数据，防止残留
> 
>     // 设置目标位置
>     aTargetPos[1] := 200.0;   // X轴目标位置 = 200 mm
>     aTargetPos[2] := 100.0;   // Y轴目标位置 = 100 mm
> 
>     // 调用直线插补准备 FB
>     fbPrepLin(
>         Position := ADR(aTargetPos), // 目标位置数组的指针
>         PositionCount := 2,          // 数组中有 2 个轴坐标
>         Velocity := fVelocity,       // 插补速度
>         Acceleration := fAccel,      // 加速度
>         Deceleration := fDecel,      // 减速度
>         BufferMode := mcAborting,    // 路径缓冲模式：新命令会中断旧命令
>         TransitionMode := mcTransModeNone, // 无过渡
>         InvokeId := 1,               // 路径段 ID (用户自定义)
>         PathData := Path             // 路径数据对象
>     );
> END_IF
> 
> // 6) 执行路径
> fbMovePath(
>     AxesGroup := AxesGroup, // 哪个轴组执行路径
>     PathData := Path,       // 要执行的路径数据对象
>     Execute := bExecute     // TRUE 时开始执行，完成后 Done = TRUE
> );
> 
> ```

### 日志

> “日志的写入我单独封装了一个 Logger 类，用的是单例模式，这样外部不会随意 new 多个实例，保证统一入口。在写日志的时候，我用了 QMutex 加锁，防止多线程同时写文件导致冲突；同时用 QTextStream 做缓存，减少磁盘 IO 次数，写入性能会更好。这样就能保证在多线程场景下，日志不会乱，也不会丢。” 



##### QTextStream

>  `QTextStream` 是 Qt 提供的文本流类，它本身带 **内存缓冲**，写入日志时可以先写入内存，等缓冲区满了调用 `flush()` 才写入磁盘。
>
>  - **减少 IO 次数**：不再每条日志都访问磁盘，而是一次性写多条，提高效率。
>  - **提高性能**：写入内存比写磁盘快很多，减少程序阻塞。
>  - **保证完整性**：结合 `QMutex` 锁，多线程写入时先写入内存缓存，再 flush，保证日志顺序不会乱。



##### **flush调用时机**

>  `QTextStream` 内部有缓冲区，当缓冲区满了，会自动 flush。
>
>  优点：不需要手动干预，系统自动保证数据写入磁盘。
>
>  适用场景：日志量大且连续写入，容忍少量延迟。



#####  日志文件自动轮转机制 

>  “考虑到系统是长时间运行的，日志不能无限写到一个文件里。我加了一个自动轮转机制， 核心逻辑就是每次写日志之前，调用 checkAndRotateLogFile() 去检查：如果日期变了，或者文件大小超过阈值，就会自动触发 createNewLogFile() 来建新文件，同时调用 cleanupOldLogFiles() 删除过期文件。这样保证日志不会无限膨胀，也不会占用太多磁盘空间。”
